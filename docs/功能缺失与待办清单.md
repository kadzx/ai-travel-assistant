# AI Travel 项目 — 功能缺失与待办清单

本文档基于当前代码库梳理**已实现**与**未实现/不完整**的功能，便于毕设答辩或后续迭代时查漏补缺。

---

## 一、项目概览

- **前端**：uni-app (Vue 3) + Pinia + uview-plus，支持 H5/多端
- **后端**：Express + MySQL + Sequelize，含 AI 对话、行程生成、社交（帖子/点赞/评论/收藏/关注）
- **核心能力**：登录注册、帖子流（推荐/关注）、发布笔记、AI 对话、AI 行程生成与保存、用户资料与主页、搜索帖子

---

## 二、已实现功能摘要

| 模块         | 功能说明 |
|--------------|----------|
| 认证         | 注册、登录、JWT、个人资料拉取与更新（含头像 base64） |
| 帖子         | 发布、列表（分页/推荐/关注/按用户/关键词/标签）、详情、点赞、收藏（单目标） |
| 社交         | 点赞、评论（含 parentId 回复）、收藏开关；关注/取关；评论列表 |
| 用户         | 个人中心、编辑资料、公开主页、关注/粉丝、我的动态（帖子/收藏/赞过） |
| AI           | 对话（历史上下文）、行程生成（目的地/天数/预算/偏好） |
| 行程         | 生成、保存、列表、详情、预览模式 |
| 上传         | 图片上传（multer） |
| 管理端接口   | 仪表盘统计、用户管理、帖子/评论管理（仅后端） |
| 搜索         | 帖子关键词、标签搜索 |

## 三、缺失或不完整功能

### 1. 通知系统 ✅ 已实现

- **实现**：后端新增 `Notification` 模型与 `GET /api/notifications`、`PUT /api/notifications/:id/read`、`PUT /api/notifications/read-all`、`GET /api/notifications/unread-count`；点赞/评论/关注时写入通知。前端通知列表页为小红书风格，支持全部已读、点击跳转帖子/用户主页。

---

### 2. 个人中心关注/粉丝数 ✅ 已对接

- **实现**：个人中心「关注」「粉丝」「获赞与收藏」已改为使用 `userStore.userInfo?.stats` 的接口数据。

---

### 3. 收藏夹（文件夹）后端已挂路由

- **实现**：已在 `socialRoutes` 挂载 `POST /api/social/favorite/folder`、`GET /api/social/favorite/folders`、`GET /api/social/favorites`。前端「收藏」Tab 仍使用现有 `GET /api/user/activities?type=favorites`；按文件夹展示与收藏时选择文件夹可作二期。

---

### 4. 景点（ScenicSpot）无前端能力

- **现状**：后端有 `ScenicSpot` 模型和 `spotRoutes`（CRUD）；前端仅在社交类型中支持 `scenic_spot`，**无**景点列表、搜索、详情页。
- **建议**：
  - 增加景点列表页（可带简单搜索/筛选）；景点详情页（展示描述、位置、票价等）；可选：帖子/评论可关联景点，或在发布时选择景点。
  - 搜索页可增加「帖子 + 景点」联合搜索或 Tab 切换。

---

### 5. 搜索仅支持帖子

- **现状**：搜索页仅调用帖子列表接口（keyword/tag），无景点、用户等维度。
- **建议**：按需扩展为「帖子 + 景点 + 用户」搜索，后端可增加统一搜索接口或分别调用现有接口在前端聚合。

---

### 6. 管理后台无前端

- **现状**：后端已有 `/api/admin/*`（仪表盘、用户状态、帖子/评论删除等）及管理员权限中间件；前端**无**任何 admin 页面。
- **建议**：若毕设需要展示「后台管理」，可做一套简单的 H5 管理端（或 PC 页）：登录后根据 role 跳转、仪表盘、用户列表、内容审核/删除等。

---

### 7. 找回密码 / 重置密码

- **现状**：仅有注册、登录；无忘记密码、邮箱验证、重置密码流程。
- **建议**：若不做邮箱，可做「手机验证码 + 重置密码」或仅预留接口与页面占位，答辩时说明为「后续扩展」。

---

### 8. 行程列表字段与接口路径 ✅ 已修复

- **实现**：前端 API 与 store 已统一为 `/itineraries`；行程列表页改用 `title`、`start_date`/`end_date` 展示，并改为小红书风格（圆角卡片、红主色 FAB、空状态引导创建）。

---

### 9. 行程入口 ✅ 已增加

- **实现**：在「我」页（个人中心）增加了「我的行程」入口卡片（小红书风格：渐变背景、地图图标、点击进入行程列表）。

---

### 10. 评论楼中楼展示

- **现状**：后端评论支持 `parent_id`，接口返回层级结构；前端帖子详情页需确认是否完整展示「回复的回复」。
- **建议**：在帖子详情评论区按 parent 树形展示，并支持「回复」某条评论时传 `parentId`。

---

### 11. 帖子隐私与 feed 过滤

- **现状**：Post 有 `privacy` 字段（如 public/private/friends）；推荐流已按 `privacy: 'public'` 过滤；「仅好友可见」是否在 feed 中按关注关系过滤需确认。
- **建议**：若需「仅好友可见」，在 getPosts 的 where 中按 `privacy` + 当前用户与作者的关注关系做过滤。

---

### 12. 用户主页「收藏」Tab 数据

- **现状**：个人中心与用户主页有「收藏」Tab，请求 `GET /api/user/activities?type=favorites`；后端已实现，返回收藏的帖子列表。
- **建议**：确认前端是否正确解析 `res`（若后端用统一包装 `{ data }`，需取 `res.data` 或与其它 Tab 一致）；若有报错或空列表，排查接口与字段映射。

---

## 四、建议优先级（毕设答辩向）

| 优先级 | 项                     | 说明 |
|--------|------------------------|------|
| 高     | 行程 API 路径与列表字段 | 否则行程功能不可用或展示错乱 |
| 高     | 个人中心关注/粉丝数对接 | 体现数据一致性 |
| 中     | 通知系统               | 提升完整度，演示效果好 |
| 中     | 行程入口               | 让答辩时能快速展示 AI 行程 |
| 中     | 收藏夹路由与前端       | 与数据模型一致，可选 |
| 低     | 景点前端、搜索扩展     | 丰富「旅行」场景 |
| 低     | 管理后台前端、找回密码 | 按答辩要求取舍 |

---

## 五、小结

- 核心链路（登录、帖子流、发布、AI 对话、行程生成与保存、用户资料）已打通，但存在**行程接口路径与列表字段不一致**、**个人中心关注/粉丝未用接口数据**等问题。
- 通知、收藏夹、景点前端、管理后台前端、找回密码等为「有模型或后端无前端」「有前端无后端」的缺口，可按上述优先级在文档中标注为「已实现 / 待实现 / 后续扩展」，便于写进毕设说明与答辩陈述。
